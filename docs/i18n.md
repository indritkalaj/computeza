# Internationalisation

> **Rule**: no user-facing string in Computeza is hardcoded in source.

Every label, message, error, log line shown to a human, button caption,
page title, and tooltip routes through the
[`computeza-i18n`](../crates/computeza-i18n) crate. English (`en`) is the
only locale today; adding new locales is a drop-in operation that requires
no code changes.

## How it works

We use [Mozilla Fluent](https://projectfluent.org/) via the
`fluent-templates` crate. Fluent is the Rust-ecosystem equivalent of
Flutter's `.arb` and is what spec section 4.1 calls for. It handles plurals,
gender, arguments, and message references natively.

Resource bundles live in `crates/<crate>/locales/<lang>/<file>.ftl` and
are embedded at compile time via the `static_loader!` macro. Air-gapped
binaries carry every shipped locale without runtime file system access.

## Adding a new string

1. Pick a kebab-case key. Group related keys with a common prefix
   (e.g. `cmd-install-*`, `err-network-*`, `pipeline-canvas-*`).
2. Add the message to the appropriate `.ftl` file under
   `crates/<owning-crate>/locales/en/`. Create a new file if no existing
   bundle is a natural home.
3. In Rust, fetch the string via the localizer:

   ```rust
   let l = computeza_i18n::Localizer::english();
   println!("{}", l.t("my-new-key"));
   ```

4. Missing keys panic in debug builds (`debug_assert!`) and render as
   `?my-new-key?` in release builds. Both are loud failure modes -- they
   exist precisely so that hardcoded strings creeping back in are caught
   immediately.

## Adding a new locale

1. Create `crates/<crate>/locales/<lang-code>/` mirroring the `en` tree.
2. Translate every `.ftl` file. Untranslated keys fall back to `en` per
   the `fallback_language: "en"` setting in `static_loader!`.
3. Rebuild. The new locale is included in the binary automatically.

No code changes are required to ship a new locale.

## Known gaps

- **Clap help text.** Clap's derive macros require `&'static str` for
  help, which forces English placeholders in `Cli` struct attributes. The
  long-term fix is to switch to clap's builder API and read help text
  through the localizer at startup. Tracked as a TODO in `crates/computeza/src/main.rs`.
- **Lint enforcement.** A `cargo xtask check-i18n` lint that scans for
  string literals reaching user-facing sinks is planned but not yet
  written. Until then, the rule is enforced socially in code review.

## Why not gettext / JSON / CSV

- **gettext** is the industry standard but its file format predates
  modern type systems and its tooling is awkward in Rust.
- **JSON** lacks first-class plurals and arguments; every team that
  starts with JSON ends up reinventing them.
- **CSV** is unmanageable beyond ~100 strings.

Fluent is purpose-built for runtime localisation, has a mature Rust
binding, and is what Mozilla ships Firefox in. The choice is captured in
spec section 4.1.
