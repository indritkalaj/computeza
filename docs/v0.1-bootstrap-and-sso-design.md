# v0.1 Design -- Auto-Bootstrap and SSO/Entra

**Status:** draft / paper architecture. No code yet.
**Author:** working document; reviewed before implementation.
**Scope:** end-to-end shape of the next milestone after v0.0.x phase 1
(install pipeline + Studio SQL editor + reconciler observe). Two
distinct architectural threads bundled here because they share the
same "post-install handshake" plumbing.

This doc supersedes the deferred-work entries in AGENTS.md for:
- "Lakekeeper bootstrap (project + warehouse + storage credentials)"
- "Garage cluster-layout bootstrap (single-node)"
- "Apply-admin-password for kanidm + grafana"
- (partially) "Secrets: scheduled auto-rotation + encrypted-mail
  notification" -- the secrets-vault changes that auto-bootstrap
  introduces are forward-compatible with auto-rotation, but rotation
  itself stays its own milestone.

---

## 1. Problem statement

### 1.1 Today (after v0.0.x phase 1)

A fresh `computeza serve` install today gets the operator to a
running data plane but stops short of a *usable* one. Four concrete
papercuts:

1. **Garage cluster is unusable until the operator hand-runs three
   CLI commands** (`gg status` -> `gg layout assign` -> `gg layout
   apply`). Until layout is applied, every Garage data operation
   returns `Internal error: Layout not ready`.
2. **Lakekeeper has no warehouse** to write Iceberg metadata into.
   The catalog browser at `/studio` shows "no warehouses
   configured" until the operator hand-runs `garage key create` +
   `garage bucket create` + `garage bucket allow` + the
   `/studio/bootstrap` form with the resulting credentials.
3. **Kanidm authenticates with an empty bearer token** because no
   admin password is applied at install time and no service-account
   reconciler token is minted. The reconciler observe loop skips
   kanidm entirely to avoid surfacing FAILED-with-wrong-reason
   (commit `2c2a692`).
4. **Operator authentication is local-only** -- `operators.jsonl`
   with argon2-hashed passwords stored on disk. No federation to
   the enterprise identity provider (Entra ID, Okta, Google
   Workspace, Keycloak, etc.) most customer organizations require.

### 1.2 What v0.1 delivers

A fresh install of v0.1 reaches steady state with **zero operator
shell commands.** The operator clicks Install on the hub,
optionally clicks "Connect identity provider" on a follow-up
screen, and the entire data plane plus identity boundary is
provisioned end-to-end.

Specifically:
- Garage cluster auto-bootstraps its single-node layout post-install.
- Garage auto-mints an access key + bucket for Lakekeeper's
  exclusive use, stores credentials in the vault, exports them in
  the `credentials.json` download.
- Lakekeeper auto-creates a project + warehouse using those
  credentials at install completion.
- Kanidm auto-mints a reconciler service-account token at install
  completion; reconciler observe transitions kanidm from UNKNOWN to
  OBSERVING.
- Grafana auto-rotates the default `admin/admin` password to a
  freshly-generated one stored in vault.
- A new `/admin/identity` page lets the operator wire Computeza ->
  Kanidm -> Entra (or another OIDC provider) as a single-screen
  bootstrap, after which the local-password login is supplemented
  (not replaced) by a "Sign in with [Entra]" button.

### 1.3 Non-goals (explicit out-of-scope for v0.1)

- **Multi-node Garage layouts.** Single-node bootstrap only;
  multi-node layout assignment is a v0.2 task that depends on the
  k8s/cloud drivers.
- **Credential rotation scheduling.** Manual rotation already works
  (commit `269abbd`); auto-rotation stays in the existing deferred
  entry.
- **PGP-encrypted mail notification of rotated credentials.** Same
  reason -- separate milestone.
- **Replacing the local-password login entirely.** v0.1 SSO is
  *additive* -- operators with valid local passwords keep working,
  operators federated via OIDC sign in with their IdP. v0.2 will
  decide whether to deprecate local passwords.
- **Per-engine OIDC authentication to Lakekeeper** (Databend/Spark
  presenting JWTs to the Iceberg REST). v0.1 keeps the engine->
  Lakekeeper layer on static credentials; engine-level OIDC trust
  is its own milestone.

## 2. Architecture: the three-layer auth model

Every conversation about auth in this codebase needs to start by
naming which layer it's about. Three distinct layers:

```
┌─────────────────────────────────────────────────────────────────┐
│ Layer 1: Operator -> Console / Lakekeeper management API        │
│   v0.0.x: argon2 local password (operators.jsonl)              │
│   v0.1:   above + OIDC SSO via Kanidm (Entra/Okta/Google upstream)│
├─────────────────────────────────────────────────────────────────┤
│ Layer 2: Engine -> Lakekeeper Iceberg REST                       │
│   v0.0.x: anonymous (localhost-only)                           │
│   v0.1:   static service-account bearer token (stored in vault)│
│   v0.2+:  OAuth2 client_credentials via Kanidm                 │
├─────────────────────────────────────────────────────────────────┤
│ Layer 3: Lakekeeper -> Garage S3                                 │
│   v0.0.x: manual access-key + secret                           │
│   v0.1:   auto-minted access-key + secret in vault             │
│   v0.x:   credential vending (Lakekeeper mints scoped temp creds│
│           per-table-read) -- only if Garage adds STS later       │
└─────────────────────────────────────────────────────────────────┘
```

This doc covers Layer 1 (the SSO half of v0.1) and Layer 3 (the
auto-bootstrap half). Layer 2 stays manual in v0.1, documented as
its own deferred entry afterward.

## 3. Auto-bootstrap design (Layer 3 + post-install handshake)

### 3.1 What "auto-bootstrap" means

Every managed component has a *steady state* that requires post-
daemon-up setup beyond the systemd unit. Auto-bootstrap is the
plumbing that runs that setup automatically, idempotently, and
visibly (with rollback paths and verbose error surfaces).

Three components need bootstrap today:

| Component | Bootstrap action | Trigger |
|---|---|---|
| Garage | Apply single-node cluster layout (`layout assign` + `layout apply`) | First install only; idempotent on re-install |
| Garage | Mint `lakekeeper` access key + bucket + bucket-allow grant | First install only; idempotent on re-install |
| Lakekeeper | Create default project + default warehouse using Garage credentials | First install only; idempotent (409 = skip) |
| Kanidm | Reset admin password (`kanidmd recover_account admin`); login; create reconciler service account; mint reconciler API token | First install only; idempotent if token already in vault |
| Grafana | Rotate `admin/admin` -> vault-stored password via Grafana's admin HTTP API | First install only; idempotent if vault entry exists |

### 3.2 Design pattern: post-install hooks

Every managed-component driver gains an **optional post-install
hook** signature:

```rust
/// Post-install bootstrap step. Runs after the systemd unit is
/// active + the daemon is accepting connections. Idempotent: must
/// no-op if the bootstrap state already exists.
///
/// Returns artifacts to be persisted by the unified install
/// orchestrator (e.g. credentials to push to the vault +
/// credentials.json export).
pub async fn post_install_bootstrap(
    root_dir: &Path,
    secrets: Option<&SecretsStore>,
) -> Result<Vec<BootstrapArtifact>, BootstrapError>;

pub struct BootstrapArtifact {
    /// Vault key: e.g. "garage/lakekeeper-key-id"
    pub vault_key: String,
    /// Secret value (may be displayed once, then redacted).
    pub value: SecretString,
    /// Human label for the credentials.json export.
    pub label: String,
    /// Whether the value should appear in the install-result page's
    /// inline credential list (false for derived non-secrets like
    /// warehouse names).
    pub display_inline: bool,
}
```

**Where it lives:** `crates/computeza-driver-native/src/linux/<component>.rs`.
The unified install orchestrator (`ui-server::finalize_managed_install`)
checks for the hook's presence per slug and invokes it.

**Why post-install, not part of install():** keeping the daemon-
start path purely about *the daemon* keeps install() reusable for
other drivers (k8s, cloud) that have their own bootstrap shapes.
Post-install hooks compose; daemon-start doesn't.

### 3.3 Cross-component handshake: Garage -> Lakekeeper

The hard case. Lakekeeper's bootstrap *needs* Garage's credentials,
which only exist after Garage's bootstrap has run. Sequencing:

```
INSTALL_ORDER (existing in COMPONENTS):
    postgres -> kanidm -> garage -> lakekeeper -> ...

Auto-bootstrap orchestration (NEW):
    1. install_service(garage)        <- daemon up
    2. garage::post_install_bootstrap <- layout + key + bucket + grant
       └─ artifacts:
            "garage/lakekeeper-key-id" = "GK..."
            "garage/lakekeeper-secret" = "..."
            "garage/lakekeeper-bucket" = "lakekeeper-default"
       └─ vault.put(each)
       └─ push_credential(each)  <- surfaces in credentials.json
    3. install_service(lakekeeper)    <- daemon up
    4. lakekeeper::post_install_bootstrap
       └─ read garage/* from vault
       └─ POST /management/v1/project   { name: "computeza-default" }
       └─ POST /management/v1/warehouse { ... storage-profile with garage creds }
       └─ artifacts:
            "lakekeeper/default-warehouse-name" = "default"  (display_inline=false)
       └─ vault.put / push_credential
```

**Failure semantics:** if Garage bootstrap succeeds but Lakekeeper
bootstrap fails (e.g. Lakekeeper schema rejects something), the
install completes with the credentials persisted, but the
install-result page shows a yellow "partial" badge with the
Lakekeeper error inline. Operator can click "Retry Lakekeeper
bootstrap" which re-runs just the Lakekeeper hook. This is the v0.1
analog to the current `/studio/bootstrap` form, which becomes a
power-user escape hatch rather than the primary path.

### 3.4 Idempotency contract

Every post-install hook must satisfy:

```
hook(state_X) -> state_Y
hook(state_Y) -> state_Y   (no error, no double-credential, no log noise)
```

How each component achieves this:

- **Garage layout:** Check `gg layout show`. If the output contains
  "Active layout" with version ≥ 1, skip layout-assign + apply.
- **Garage key:** Check `gg key info lakekeeper`. If it returns
  valid data (key exists), skip key-create. **Edge case:** Garage
  CLI redacts the secret on `key info` -- we can't re-read it. If
  the vault has the secret, trust it. If the vault is missing the
  secret BUT the key exists in Garage, the install reports a
  recovery error and the operator manually rotates via the existing
  rotate handler.
- **Garage bucket / allow:** Similarly check existence first. The
  `bucket allow` is idempotent on the Garage side (re-running just
  re-asserts the grant).
- **Lakekeeper project / warehouse:** Already handled -- project
  creation returns 409 if already exists, warehouse creation
  similarly. Existing `run_lakekeeper_bootstrap` treats 409 as
  success.

### 3.5 Credentials JSON export integration

The existing `/install/credentials.json/{job_id}` endpoint
(commit shipped earlier) already aggregates `push_credential(...)`
calls. Auto-bootstrap simply pushes more entries:

```json
{
  "format": "computeza-credentials-v1",
  "issued_at": "2026-05-14T09:30:00Z",
  "credentials": [
    {
      "component": "postgres",
      "label": "superuser password",
      "username": "postgres",
      "value": "<hex>",
      "secret_ref": "postgres/admin-password"
    },
    {
      "component": "garage",
      "label": "Lakekeeper-scoped Access Key ID",
      "username": null,
      "value": "GK029bcc02bc15f7ab9e12980d",
      "secret_ref": "garage/lakekeeper-key-id"
    },
    {
      "component": "garage",
      "label": "Lakekeeper-scoped Secret Access Key",
      "username": null,
      "value": "<64-char-hex>",
      "secret_ref": "garage/lakekeeper-secret"
    },
    {
      "component": "lakekeeper",
      "label": "Default warehouse name",
      "username": null,
      "value": "default",
      "secret_ref": null
    }
  ]
}
```

The download is **one-shot** (per existing behavior) -- the operator
either saves it or risks losing it. The vault retains everything;
this download is for offline backup / handoff.

### 3.6 Failure recovery

Three failure modes worth thinking about:

| Failure | Recovery path |
|---|---|
| Garage daemon up but layout-bootstrap fails (network, CLI signature drift) | Install marks Garage component as "started but not provisioned"; the existing `/studio/bootstrap` form becomes the recovery UI. Operator hand-runs the layout commands per the help text, then re-runs the wizard. |
| Garage credentials persisted but Lakekeeper bootstrap fails (schema drift) | Lakekeeper component marked "started but warehouse missing". The `/studio/bootstrap` form is pre-filled with the vault-stored credentials. One click to retry. |
| Vault unavailable at bootstrap time (passphrase wrong, file corrupt) | Bootstrap aborts; the credentials never get persisted. Operator sees a hard error on the install-result page with the actionable fix ("verify COMPUTEZA_SECRETS_PASSPHRASE or remove the corrupt vault file"). |

## 4. SSO/Entra ID design (Layer 1)

### 4.1 The end-state operator experience

After v0.1 ships:

1. Fresh install completes (auto-bootstrap green).
2. Operator clicks `/admin/identity -> Connect identity provider`.
3. Wizard asks for the provider type (Entra, Okta, Google, Keycloak,
   Generic OIDC).
4. Operator pastes their OIDC discovery URL + client ID + client
   secret (or runs the wizard's Entra-app-registration helper that
   pre-fills them via Microsoft Graph).
5. Computeza tests the connection by initiating a discovery fetch
   and validating the JWKS.
6. On success, Kanidm is configured as an OIDC client of the
   upstream provider. Computeza, Lakekeeper, and (later) Databend
   become OIDC clients of Kanidm.
7. The login page shows both **"Sign in with local password"** and
   **"Sign in with [Entra]"** buttons. Either works.

### 4.2 Identity dataflow

```
User (Indrit) ──┐
                │  click "Sign in with Entra"
                ▼
        Computeza /auth/login ──┐
                                │  HTTP 302 -> Kanidm /oauth2/authorize?...
                                ▼
                       Kanidm OIDC IdP ──┐
                                         │  HTTP 302 -> Entra /oauth2/v2.0/authorize
                                         ▼
                                   Entra ID ──┐
                                              │  user authenticates with Entra
                                              ▼
                                   Entra ID returns ID token to Kanidm
                                              │
        Kanidm validates Entra token,        │
        looks up or auto-provisions the      │
        user in its local user DB,           │
        issues its own JWT,                  │
        302 back to Computeza /auth/callback │
                                              ▼
        Computeza validates Kanidm JWT via JWKS,
        creates session cookie, redirects to /

  Subsequent API calls:
        Computeza presents Kanidm JWT to Lakekeeper /management/v1/*
        Lakekeeper validates via OIDC discovery (Kanidm)
        Permission decisions made against Lakekeeper's own RBAC
        keyed by the `sub` (subject) or `oid` (Entra object ID) claim
```

### 4.3 Bootstrap pieces (each its own commit)

| # | Step | Where | Effort |
|---|---|---|---|
| 1 | Kanidm post-install bootstrap: `recover_account` -> login -> service-account creation -> API token mint | `crates/computeza-driver-native/src/linux/kanidm.rs` | ~2-3 days |
| 2 | `/admin/identity` page (Computeza UI) -- discovery-URL + client-id + client-secret form for the upstream IdP | `crates/computeza-ui-server/src/lib.rs` | ~2 days |
| 3 | Kanidm OIDC-client config writer -- programmatically register the upstream IdP in Kanidm via its admin API | new helper in `computeza-ui-server` + `computeza-reconciler-kanidm` | ~2-3 days |
| 4 | OIDC client middleware for Computeza -- JWKS fetch, ID token validation, session cookie issuance | new module `crates/computeza-ui-server/src/oidc.rs` | ~3-5 days |
| 5 | Login page changes: render "Sign in with [provider name]" button conditional on whether identity is wired | `ui-server` template | ~1 day |
| 6 | Lakekeeper OIDC trust: set `LAKEKEEPER__OPENID_PROVIDER_URI` env var pointing at Kanidm; restart unit | `crates/computeza-driver-native/src/linux/lakekeeper.rs` env section | ~1 day |
| 7 | Audit log enrichment: every audit entry carries the OIDC subject claim instead of local username | `crates/computeza-audit` + everywhere that calls `audit::log(...)` | ~1-2 days |
| 8 | Operator-management page changes: group membership now maps Entra group claims -> Computeza permissions | `ui-server::auth` + `/admin/operators` page | ~2 days |

**Total: ~14-21 days of focused work.** This is the larger half of
v0.1.

### 4.4 What about provider-specific concerns

- **Entra ID:** the wizard offers an automated app-registration
  helper that uses the operator's Microsoft Graph credentials to
  create the OIDC client app on the operator's behalf. Optional --
  operators can paste pre-existing client credentials. The reason
  to automate: Entra app-reg requires ~6 clicks in the Azure portal
  and is the #1 source of operator setup errors at competing
  products.
- **Okta:** standard OIDC; no special wizard. Operator follows the
  generic flow.
- **Google Workspace:** OIDC with the `hd` claim restriction for
  domain-scoping. The wizard surfaces a "Restrict to Google domain"
  field.
- **Keycloak:** OIDC; works with the generic flow. Operators
  running on-prem Keycloak get an additional "trust internal-CA TLS
  cert" toggle.
- **Generic OIDC:** discovery URL + client credentials, no
  provider-specific UI.

### 4.5 Local password during SSO transition

Two strategies:

- **(a) Side-by-side:** both login methods always work. Operators
  can use local password for break-glass even when SSO is wired.
  **Pro:** safe, no lockout risk. **Con:** local password becomes a
  durable attack surface.
- **(b) SSO-only with break-glass:** local password disabled when
  SSO is wired, except for one operator marked "break-glass" whose
  local password remains active. **Pro:** smaller attack surface.
  **Con:** discovering the break-glass operator's password file is
  effectively root-equivalent.

**Recommendation: (a) for v0.1**, with a v0.2 admin toggle to
switch to (b). Most enterprise customers will run for months in
(a) before they trust SSO enough to disable local passwords.

### 4.6 The "Sign in with Entra" button -- surface design

The current login page is rendered by `ui-server::lib.rs::render_login_page`.
After identity is wired, the page rendering checks
`is_oidc_configured()` and conditionally renders the SSO button.
The button POSTs to `/auth/oidc/login` which initiates the OAuth2
flow.

The button is *not* the only login option -- local-password form
stays visible underneath, as a "Sign in another way" affordance.

## 5. Migration path from v0.0.x

A v0.0.x install upgraded in-place to v0.1:

1. **No-op for unchanged components.** Postgres, Databend, Qdrant,
   etc. don't gain bootstrap hooks; their install behavior is
   identical.
2. **Garage gains layout-bootstrap.** On `computeza` binary upgrade
   + first restart, the auto-bootstrap helper detects that layout
   is already applied (operator did it by hand following the v0.0.x
   guide) and no-ops.
3. **Garage gains key-bootstrap.** Detects that the `lakekeeper`
   key already exists in Garage (operator created it by hand) and
   no-ops. **Caveat:** the secret is lost (can't re-read from
   Garage). The operator must rotate via the existing rotation flow
   to repopulate the vault.
4. **Lakekeeper gains warehouse-bootstrap.** Detects that a
   warehouse named `default` exists, no-ops.
5. **SSO is opt-in.** Existing v0.0.x installs continue with local
   password login. Operators visit `/admin/identity` when they
   want to wire SSO.

**Estimated migration time per existing install: <60 seconds**
(the auto-bootstrap idempotency checks are fast network calls).

## 7. Open questions to decide before building

These are real decisions the doc doesn't pre-commit. Flag each
with a recommended default in italics, but they're up for review.

1. **Auto-bootstrap rollback semantics.** If Garage bootstrap
   succeeds and Lakekeeper bootstrap fails, do we roll back Garage
   (delete the key + bucket)?
   *Default: no rollback. The credentials are still in the vault;
   the operator can retry Lakekeeper bootstrap separately.*

2. **Kanidm reconciler-token TTL.** Kanidm API tokens can be
   issued with expiry. What TTL do we choose for v0.1?
   *Default: no expiry (long-lived). v0.2 introduces a token-
   rotation hook that re-mints quarterly.*

3. **SSO group -> permission mapping.** When an Entra user signs in,
   their `groups` claim contains Entra group OIDs. Do we map these
   to Computeza permissions automatically, or require the operator
   to define mappings explicitly?
   *Default: explicit mappings on `/admin/operators`. Auto-mapping
   is too magical and causes silent privilege escalation when
   Entra groups change.*

4. **OIDC scope claims.** What claims do we require from the
   upstream IdP for a valid login? `email`? `email_verified`?
   `oid`? `name`?
   *Default: `sub` + `email` minimum. `name` for display only.
   `email_verified=true` enforced. `oid` (Entra-specific) stored
   if present.*

5. **Provider-config storage location.** Where does the OIDC
   client_secret live -- vault, or separate state-DB row?
   *Default: vault, behind `identity/oidc-client-secret`. Same
   protection as any other credential.*

6. **First-operator bootstrap when SSO is wired.** If an empty
   `operators.jsonl` install gets SSO wired before any operator is
   created, the first SSO sign-on must be auto-promoted to admin.
   How do we make this safe?
   *Default: if `operators.jsonl` is empty AND SSO is wired AND the
   incoming user's email matches a "designated bootstrap admin"
   email entered during identity setup, auto-promote. Otherwise
   reject with "no operator account found; ask an admin to invite
   you".*

7. **Credentials.json download -- display vs persistence.** The
   `value` field of each credential in `credentials.json` is the
   plaintext secret. Should it be shown in the operator-console UI
   the first time (one-shot show) or only available via download?
   *Default: download-only. The console UI never displays
   plaintext secrets except on the rotation-result page, which
   already exists.*

8. **Multi-tenancy implications.** When per-tenant isolation
   lands (the deferred tenant-selector entry), do per-tenant
   service accounts to Kanidm get auto-minted? Or is the v0.1
   single-tenant default OK?
   *Default: single-tenant for v0.1. Multi-tenant SSO is a
   v0.2 concern.*

## 9. Risks + mitigations

| Risk | Mitigation |
|---|---|
| Garage CLI signature drifts (e.g., `key create --name` -> positional, which already happened in 0.9+) | Bootstrap helper invokes with `--help` first to detect supported flags; abort with clear error on mismatch. Pin Garage version in our installer; bump deliberately. |
| Lakekeeper schema drift across releases | Pin Lakekeeper version. The existing verbose-error surface in `run_lakekeeper_bootstrap` already shows the response body verbatim, enabling fast iteration. |
| Kanidm `recover_account` behavior changes between versions | Pin Kanidm version. The bootstrap helper uses the CLI, not the HTTP API directly, so we depend on a stable CLI surface (kanidmd CLI is more stable than the API across releases). |
| OIDC discovery URL unreachable at sign-in time (network partition between Computeza and Kanidm, or Kanidm and Entra) | JWKS cached for 24h after first fetch. Login page falls back to local-password form when SSO is unhealthy. |
| Vault passphrase lost during upgrade | The vault remains encrypted; the operator must re-enter the passphrase. The auto-passphrase generation (existing) writes the passphrase to a 0600 file; document recovery clearly. |
| Operator wires SSO incorrectly and locks themselves out | Local-password login (option 4.5(a)) remains available; the operator signs in with their local password and reconfigures SSO. |
| Entra app-registration helper introduces a new dependency on the operator's Microsoft Graph credentials | Optional path. The manual paste-discovery-URL flow remains primary. |

## 10. Success criteria

v0.1 ships when all of these hold on a fresh Ubuntu 24.04 WSL2 install:

- [ ] `computeza install all` completes with all 10 components green (postgres, kanidm, garage, lakekeeper, databend, qdrant, restate, greptime, grafana, openfga; xtable stays scaffold-only per existing constraint).
- [ ] `/status` shows OBSERVING for all 10 (kanidm transitions out of UNKNOWN because the reconciler-token was auto-minted).
- [ ] `/studio` catalog pane shows a `default` warehouse with no operator intervention.
- [ ] `/studio` SQL editor's autocomplete suggests the warehouse name without manual config.
- [ ] `/install/job/{id}/credentials.json` downloads a JSON file containing 4+ credential entries (postgres password, garage key-id+secret, kanidm reconciler-token, grafana admin-password) with the right `secret_ref` keys.
- [ ] Operator can wire Entra ID via `/admin/identity` in <5 minutes (with pre-existing Entra tenant access).
- [ ] After SSO is wired, an Entra-authenticated operator can sign in via the SSO button, perform an install action, and have it audit-logged with their Entra `oid` claim.
- [ ] Existing v0.0.x install upgrades to v0.1 binary without operator intervention; idempotency tests pass.
- [ ] All security warnings (vault-passphrase, SSO-misconfig, missing-credentials) surface inline in the install-result page with actionable copy.
